## Follow-up Investigation

### Duplicate command output in Claude UI

- Symptom:
  - The same response appeared once in the Bash tool block and once again as a final assistant bubble
- Root cause:
  - The command wrapper asked Claude to run Bash and then effectively echo the script output again
  - `scripts/parent.py` was already printing the final user-facing response to stdout
- Fix:
  - Updated `.claude/commands/parent.md`
  - Updated `.claude/commands/parent-no-opus.md`
  - Updated `scripts/install_global_commands.py`
  - Wrapper instructions now explicitly require no final assistant message after the Bash tool finishes

### Latency reduction analysis

- Baseline conclusion:
  - The dominant cost is architectural because `/parent` uses one Claude pass to interpret the slash command and a second Claude pass inside `scripts/parent.py`
- Biggest avoidable delay:
  - The wrapper was forcing `scripts/parent.py` to recover command arguments from the Claude transcript file
  - That path uses retry polling and can add noticeable latency or long worst-case stalls
- Optimization applied:
  - Updated both command wrappers and the global installer to pass `$ARGUMENTS` directly to `scripts/parent.py` through stdin
  - This preserves quoting safely via a here-doc and lets `load_request_text()` return immediately without transcript polling when arguments are present
- Remaining latency:
  - A second Claude invocation is still required by design, so `/parent` will remain slower than a normal direct Claude reply
